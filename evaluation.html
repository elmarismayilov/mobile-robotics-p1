02 // CRITICAL ANALYSIS

Successes & Failures

Operational Wins

  • Dynamic Obstacle Avoidance: The DWA Local Planner reacted instantly to human movement in the path.
  • LiDAR Precision: Stable detection of thin obstacles (table legs) thanks to the LDS-02 sensor.
  • Compute Offloading: Running SLAM on a Host PC prevented Pi CPU thermal throttling.
  • Transparent Link: ROS2 DDS handled the distributed node communication with < 5ms latency.

Observed Limitations

  • Wi-Fi Dependency: System failure occurred if the Host PC lost connection to the robot's local network.
  • Time Sync: Clock desynchronization between Pi and PC occasionally caused TF transform aging errors.
  • Lidar/Glass: Standard LiDAR failure to detect transparent obstacles remained a sensor-level bottleneck.
03 // DESIGN TRADE-OFFS

The Balance of Engineering

Reflecting on the technical sacrifices made to ensure system stability and real-time response.

Resolution vs. Speed

We chose a 0.05m grid resolution. Lower values improved mapping detail but caused the global planner to time out on the Raspberry Pi 4.

Host vs. Edge

We offloaded 90% of compute to the Host PC. This allowed for complex SLAM tuning that would be impossible on the Pi standalone.

Frequency vs. Bandwidth

LiDAR scan rate was set to 5Hz to ensure Wi-Fi bandwidth wasn't saturated by the peer-to-peer DDS traffic.

04 // SYSTEM RELIABILITY

Data Quality & Integrity

Interpreting the logs: How reliable is the generated occupancy grid under stress?

Packet Loss

Occasional drops in Wi-Fi transmission caused transient flickering in LIDAR scans. The system relies on the Occupancy Grid accumulation to maintain a stable world model despite these momentary data gaps.

[WARN] Network: Transient packet loss detected
[INFO] Map status: Persisting existing costmap data

Sensor Noise

Electrical noise from the motors slightly impacted IMU gyro readings. The EKF (Extended Kalman Filter) effectively suppressed these high-frequency spikes.
